# Diagnose（系统化调试）

**调试纪律：跳过阶段必须有明确理由。**

## Phase 1 — 构建反馈循环

**这是技能本身。其他都是机械的。**

如果你有一个快速、确定性、可Agent运行的pass/fail信号，你就会找到原因——二分法、假设测试和插桩都只是消耗那个信号。如果你没有信号，看再多代码也救不了你。

在这里投入不成比例的努力。**要激进。要有创意。拒绝放弃。**

### 构建反馈循环的方法（按优先级）

1. **失败测试** 在任何触及bug的边界 — 单元、集成、e2e
2. **Curl / HTTP 脚本** 对运行中的开发服务器
3. **CLI 调用** 配合fixture输入，diff stdout对已知良好快照
4. **无头浏览器脚本** (Playwright/Puppeteer) — 驱动UI，断言DOM/console/network
5. **重放捕获的trace** 保存真实网络请求/载荷/事件日志到磁盘，隔离重放
6. **一次性harness** 启动系统的最小子集（一个服务，mocked deps）用单个函数调用练习bug代码路径
7. **属性/模糊循环** 如果bug是"有时输出错误"，运行1000个随机输入寻找失败模式
8. **二分harness** 如果bug出现在两个已知状态之间（commit、数据集、版本），自动化"在状态X启动，检查，重复"以便`git bisect run`
9. **差分循环** 用旧版本vs新版本（或两个配置）运行相同输入并diff输出
10. **HITL bash脚本** 最后一招。如果必须人工点击，用`scripts/hitl-loop.template.sh`驱动他们使循环仍然是结构化的

构建正确的反馈循环，bug就解决了90%。

### 迭代循环本身

把循环当作产品。一旦有了_一个_循环，问：
- 能让它更快吗？（缓存设置，跳过无关初始化，缩小测试范围）
- 能让信号更尖锐吗？（断言特定症状，不只是"没崩溃"）
- 能让它更确定性吗？（固定时间，种子RNG，隔离文件系统，冻结网络）

30秒的flaky循环和没有循环差不多。2秒的确定性循环是调试超能力。

### 非确定性bug

目标是清洁的复现而是**更高的复现率**。循环触发100×，并行化，加压，缩小时间窗口，注入sleep。50%的flake bug是可调试的；1%不是——不断提高比率直到可调试。

### 当真的无法构建循环时

明确说出来。列出你尝试过的。请求用户提供：(a) 能复现的环境，(b) 捕获的artifacts (HAR文件、日志dump、core dump、带时间戳的屏幕录制)，或 (c) 添加临时生产插桩的许可。**不要**在没有循环的情况下继续假设。

### 验证

继续Phase 2前：
- [ ] 你有一个你相信的循环

## Phase 2 — 复现

运行循环。看bug出现。

确认：
- [ ] 循环产生用户描述的失败模式——不是恰好在附近的另一个失败。错误的bug = 错误的修复。
- [ ] 失败在多次运行中可复现（或者对于非确定性bug，以足够高的速率可复现以供调试）
- [ ] 你已捕获确切症状（错误消息、错误输出、慢时序）以便后续阶段验证修复确实解决了问题

在复现bug前不要继续。

## Phase 3 — 假设

测试前生成**3-5个排名假设**。单一假设生成会锚定在第一个看似合理的想法上。

每个假设必须是**可证伪的**：陈述它做出的预测。

> 格式："如果<X>是原因，那么<改变Y>会使bug消失 / <改变Z>会使它更糟"

如果你不能陈述预测，那个假设只是一种感觉——丢弃或加深它。

**在测试前向用户展示排名列表。** 他们通常有领域知识能立即重新排名（"我们刚部署了#3的变更"），或知道你已经排除的假设。便宜的检查点，大的时间节省。如果用户不在，不要阻塞——如果用户AFK，按你的排名继续。

## Phase 4 — 插桩

每个探测必须映射到Phase 3的特定预测。**一次只改一个变量。**

工具偏好：
1. **调试器/REPL检查** 如果环境支持。一个断点胜过十个日志。
2. **针对性日志** 在区分假设的边界。
3. 永远不要"日志一切然后grep"。

**标记每个debug日志** 用唯一前缀，如`[DEBUG-a4f2]`。最终清理变成单一grep。未标记的日志保留；标记的日志删除。

**性能分支**：对于性能回归，日志通常是错的。替代：建立基线测量（时序harness、`performance.now()`、profiler、查询计划），然后二分。**先测量，再修复。**

## Phase 5 — 修复 + 回归测试

在修复前写回归测试**但仅当有正确的边界**时。

正确的边界是测试作为发生在调用点的**真实bug模式**的边界。如果唯一可用的边界太浅（单调用者测试而bug需要多个调用者，无法复制触发bug的链的单元测试），那里的回归测试给出虚假信心。

**如果没有正确的边界，那本身就是发现。** 注意它。代码库架构阻止bug被锁定。标记给下一阶段。

如果有正确的边界：
1. 将最小化repro变成该边界的失败测试
2. 看它失败
3. 应用修复
4. 看它通过
5. 对原始（非最小化）场景重新运行Phase 1循环

## Phase 6 — 清理 + 复盘

宣布完成前必须：
- [ ] 原始repro不再复现（重新运行Phase 1循环）
- [ ] 回归测试通过（或记录了边界缺失）
- [ ] 所有`[DEBUG-...]`插桩已删除（grep该前缀）
- [ ] 一次性原型已删除（或移到明确标记的调试位置）
- [ ] 正确的假设在commit/PR消息中陈述——以便下一个调试者学习

**然后问：什么能防止这个bug？** 如果答案涉及架构变更（没有好的测试边界、纠缠的调用者、隐藏耦合），用具体细节交递给`/improve-codebase-architecture`技能。在修复进入后提出建议——不是之前——你现在有更多信息。
